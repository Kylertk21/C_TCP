## TUN/TAP Devices

TAP devices are used to intercept low-level network traffic from the linux kernel.
They are special character devices that act as the factory for creating TUN/TAP interfaces.
TUN/TAP devices are used by networking userspace applications to manipulate L3/L2 traffic.
TAP devices are layer 2 (ethernet frames with MAC headers), 
TUN devices are layer 3 (IP packets, no ethernet header)
Common example is tunneling, where a packet is wrapped inside the payload of another packet.

TUN/TAP instantiation:

    int tun_alloc(char *dev) {
        /* Takes a desired interface name, returns an fd
         * than can send and receive packets, if dev is empty,
         * kernel picks a name
         */
       
        struct ifreq ifr; // standard linux struct for passing
                          // interface config to kernel via ioctl
        int fd, err;

        if (( fd = open("/dev/net/tap", O_RDWR)) < 0) { // open tun/tap interface, 
                                                        // gives generic handle
            perror("Cannot open TUN/TAP dev");
            exit(1);
        }

        CLEAR(ifr); // Zero out struct

            /* Flags: IFF_TUN   - TUN device (no Ethernet headers)
             *        IFF_TAP   - TAP device
             *
             *        IFF_NO_PI - Do not provide packet information (4-byte)
             */

        ifr.ifr_flags = IFF_TAP | IFF_NO_PI;
        if ( *dev ) { // Check if first character isnt null (string is full)
            strncpy(ifr.ifr_name, dev, IFNAMSIZ); // Assign 16, max interface name length
        }

        if ( (err = ioctl(fd, TUNSETIFF, (void *) &ifr)) < 0) {
        /* Actually create the interface, ioctl sends a command to the kernel
         * through the file descriptor. TUNSETIFF tell TUN/TAP driver to bind
         * fd to a specific interface with the config in ifr
         */
            perror("ERR: Could not ioctl tun: %s\n", strerror(errno));
            close(fd);
            return err;
        }

        /* New network interface tap0 will appear on the system, visible via ip link
         * any data written to fd appears as incoming packets on that interface
         * any packets sent out through that interface can be read from fd
         */

        strcpy(dev, ifr.ifr_name);
        return fd;
        
        /* kernel fills ifr.ifr_name with actual assigned name, writes back to 
         * caller's dev buffer and returns the file descriptor
         */

    }


    int main(void) {
      char dev[IFNAMSIZ] = "test"; // writable buffer on the stack
      int fd = tun_alloc(dev);
      if (fd < 0) {
        fprintf(stderr, "Failed to allocate TAP device\n");
        return 1;
      }

      if (ioctl(fd, TUNSETPERSIST, 1) < 0) { // set interface to persist
        perror("ERR: Could not set persistent");
        close(fd);
        return 1;
      }

      printf("TAP device created: %s (fd=%d)\n", dev, fd);
      close(fd);
      return 0;

    }

## Ethernet Frame Format

frame header declaration:

    #include <if_ether.h> // maps ethertypes and hex values

    struct eth_hdr {
        unsigned char dmac[6]; // destination mac address
        unsigned char smac[6]; // source mac address
        uint16_t ethertype; // 2 octet, indicates either length or payload type
                            // > 1536 = type of payload
        unsigned char payload[]; // pointer to eth frame's payload
                                 // contains ARP / IPv4 packet this time
                                 // Pad bytes are appended if < 48 bytes
    } __attribute__((packed)); // instruct GCC to not optimize memory layout

    struct eth_hdr *hdr = (struct eth_hdr *) buf;
    // need packed for this type of declaration
    // can also serialize protocol data manually if padding bytes
    // are needed

### Address Resolution Protocol

dynamically maps 48-bit ethernet address (MAC address) to a protocol address
(IPv4 address). Several layer 3 protocols can be used like IPv4 or CHAOS
Used to broadcast and query network for a service

    struct arp_hdr {
        uint16_t hwtype; // 2 octet, determines link layer type, eth in this case, 90x0001
        uint16_t protype; // indicates protocol type , IPv4 in this case, 0x0800
        unsigned char hwsize; // 1 octet, contain size of hardware
        unsigned char prosize; // and protocol fields, 6B for MAC, 4B for IP
        uint16_t opcode; // type of ARM message, request (1) / reply (2) / RARP request (3) / RARP reply (4)
        unsigned char data[]; // payload of ARP message, IPv4 info below in this case
    } __attribute((packed));


    struct arp_ipv4 {
        unsigned char smac[6];
        uint32_t sip; // sender IP address
        unsigned char dmac[6];
        uint32_t dip; // receiver IP address
    } __attribute__((packed));


#### Address Resolution Algorithm

?Do I have the hardware type in ar$hrd?
Yes: (almost definitely)
  [optionally check the hardware length ar$hln]
  ?Do I speak the protocol in ar$pro?
  Yes:
    [optionally check the protocol length ar$pln]
    Merge_flag := false
    If the pair <protocol type, sender protocol address> is
        already in my translation table, update the sender
        hardware address field of the entry with the new
        information in the packet and set Merge_flag to true.
    ?Am I the target protocol address?
    Yes:
      If Merge_flag is false, add the triplet <protocol type,
          sender protocol address, sender hardware address> to
          the translation table.
      ?Is the opcode ares_op$REQUEST?  (NOW look at the opcode!!)
      Yes:
        Swap hardware and protocol fields, putting the local
            hardware and protocol addresses in the sender fields.
        Set the ar$op field to ares_op$REPLY
        Send the packet to the (new) target hardware address on
            the same hardware on which the request was received.
